Why I still write plain CSS Modules

Every few years the front end agrees on a new way to write styles, and every few years the old way turns out to still work. This blog has no utility framework in it. It has design tokens, CSS Modules, and about two hundred lines of CSS that I can read end to end.

Tokens are the actual contract

A component never picks a colour. It asks for a role — --color-muted-foreground, --color-primary — and the theme decides what that means today. Swap the accent and nothing in the component changes: 

.tag {
  color: var(--color-primary);
  font-family: var(--font-mono);
  font-size: var(--typography-mono-size);
}

That is the whole trick. The linter enforces it — a raw hex value in a CSS Module fails the build — so the rule cannot quietly rot.

What the levels mean

  • Primitives — the raw ramp. An implementation detail; components never touch it.
  • Derived — the accent families, one per data-accent.
  • Semantic — the roles. The only level anything else is allowed to read.

A design system is not a component library. It is the set of decisions you have agreed to stop making.

The parts that took the longest

  1. Getting the dark surfaces to sit apart without a drop shadow.
  2. Making the focus ring visible on every accent — see the notes on contrast.
  3. Deciding that the empty state must never own an h1.

None of that is framework work. It is just deciding, once, and then writing it down where the next person — usually me — will find it.
