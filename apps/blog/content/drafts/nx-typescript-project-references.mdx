I added a small package to this repo — the plugin that posts to Slack when a
release goes out — and the build went red. Not on a type error: nothing I wrote
was wrong. It failed with this.

```
NX   The workspace is out of sync

[@nx/js:typescript-sync]: Some TypeScript configuration files are missing project
references to the projects they depend on.
```

The new package imports another one, `@dev-blog/notify`. My code said so. The
dependency was real and correct. What was missing is that a **second file** also
had to say so, and did not. This is about that second file: what it is, why it
exists, and why I stopped writing it by hand.

## What a project reference is

By default `tsc` treats a codebase as one pile of files and typechecks the whole
thing at once. That is fine until the codebase is a monorepo — here, an app and a
handful of libraries — and you want two things a single pile cannot give you:
build the parts in the right order, and rebuild only the part that changed.

Project references are TypeScript's answer. You split the code into **projects**,
each with its own `tsconfig`, and each one declares which other projects it
depends on:

```json
{
  "references": [{ "path": "../../libs/notify" }]
}
```

Turn on `composite: true` and TypeScript will build them as a graph: `notify`
before the package that references it, its output cached so an unchanged project
is not compiled twice. My base config turns it on for everything:

```json
{
  "compilerOptions": {
    "composite": true,
    "customConditions": ["@dev-blog/source"]
  }
}
```

That second line is the part I would keep even in a much smaller repo. Each
package's `package.json` carries an export condition pointing at its source:

```json
"exports": { ".": { "@dev-blog/source": "./src/index.ts" } }
```

and `customConditions` tells TypeScript to honour it. So when the release plugin
imports `@dev-blog/notify`, TypeScript resolves it to `notify`'s **source**, not
to a built `.d.ts` in `node_modules`. Cross-package go-to-definition lands on the
real line. A rename propagates across packages. Types are accurate before
anything has been built. The monorepo reads like one codebase instead of a
handful of published libraries.

## Why it drifts

The catch is in the reference list. There are now two places that record the same
fact. The **import** says the release plugin depends on `notify`. The
**reference** in its tsconfig has to say it too. One is the code you write; the
other is a bookkeeping file next to it.

Keep two records of one fact in sync by hand and they drift. You add an import
and forget the reference — which is exactly what I did. Nothing tells you at the
time, because the import resolves fine. It bites later, out of order: the
incremental build compiles a package against a stale declaration of one it
depends on, or builds them in the wrong order, and the error you get points
anywhere but at the line you forgot.

## Why a tool writes it now

Here is the thing that makes the whole problem dissolve: **nx already knows the
graph.** It has to. It cannot run `build` or `typecheck` in the right order
without first working out that the release plugin depends on `notify`. The
dependency graph is not something I should be re-typing into a tsconfig by hand —
it is already computed, sitting right there.

So `@nx/js:typescript-sync` reads that graph and writes the references to match
it. `nx sync` and the tsconfigs are correct. And — this is the part that turned
my build red — it also **gates**: if the references on disk do not match the
graph, the build fails before it runs, with the "out of sync" message above.

That is what stopped me. I had added `@dev-blog/release`, imported
`@dev-blog/notify`, and left the references untouched. nx knew the two disagreed
and refused to build a drifted config. One command fixed it:

```
$ nx sync
```

It wrote two lines — a reference to the new plugin in the root tsconfig, and a
reference to `notify` inside the plugin's — and the build went green.

## Where it lands

I have a rule on this blog by now: a fact the code already contains should not be
re-entered by a human, because a human will get it wrong and no one will notice.
The last time it was icon colours in a generated file nobody reads. This is the
same shape. The dependency graph is a fact the imports already contain. The
project references are that fact, copied. So the copy is generated, and a gate
fails the build when the copy and the original disagree — which is the only way I
was ever going to catch a reference I forgot.

The honest cost: project references earn their keep at scale, and this repo is not
large. For a few libraries the incremental-build win is mostly theoretical. What
is not theoretical, even here, is the source resolution — `customConditions`
pointing every import at the real source file. That one I would set up on day one,
in a repo of any size. The references are the bookkeeping it drags along, and the
right amount of attention to pay bookkeeping is none: let the graph write it.
