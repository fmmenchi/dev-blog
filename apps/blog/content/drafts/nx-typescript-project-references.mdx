I added a small package to this repo — the plugin that posts to Slack when a
release goes out — and the build went red. Not on a type error: nothing I wrote
was wrong. It failed with this.

```text
NX   The workspace is out of sync

[@nx/js:typescript-sync]: Some TypeScript configuration files are missing project
references to the projects they depend on, contain stale project references, or
have duplicate project references.
```

The new package imports another one, `@dev-blog/notify`. My code said so, and the
exports map that makes the import resolve — [the subject of part 1](/blog/can-i-import-this) —
was in place too. What was missing is that a **second file** also had to say so,
and did not. This is about that second file, and about why I
stopped writing it by hand.

## What a project reference is

By default `tsc` treats a codebase as one pile of files and typechecks the whole
thing at once. That is fine until the codebase is a monorepo — here, an app and a
handful of libraries — and you want two things a single pile cannot give you:
build the parts in the right order, and rebuild only the part that changed.

Project references are TypeScript's answer. You split the code into **projects**,
each with its own `tsconfig`, and each one declares which other projects it
depends on:

```json
{
  "references": [{ "path": "../../libs/notify" }]
}
```

Turn on `composite: true` and run `tsc --build`, and TypeScript treats them as a
graph: `notify` before the package that references it, its output cached so an
unchanged project is not compiled twice. Plain `tsc` gets you neither — the
ordering and the reuse are what
[build mode](https://www.typescriptlang.org/docs/handbook/project-references.html)
adds. My base config turns composite on for everything:

```json
{
  "compilerOptions": {
    "composite": true,
    "customConditions": ["@dev-blog/source"]
  }
}
```

The second line has nothing to do with project references. It is how a workspace
import lands on a package's own source instead of a built `.d.ts`, which is the
subject of [part 1](/blog/can-i-import-this). The two settings share a file
because both are workspace-wide, and that is all they share.

## Why it drifts

The catch is in the reference list. There are now two places that record the same
fact. The **import** says the release plugin depends on `notify`. The
**reference** in its tsconfig has to say it too. One is the code you write; the
other is a bookkeeping file next to it.

Keep two records of one fact in sync by hand and they drift. You add an import
and forget the reference — which is exactly what I did. Nothing tells you at the
time, because the import resolves fine. It bites later, out of order: the
incremental build compiles a package against a stale declaration of one it
depends on, or builds them in the wrong order, and the error you get points
anywhere but at the line you forgot.

## Why a tool writes it now

**nx already knows the graph.** It has to: it cannot run `build` or `typecheck`
in the right order without first working out that the release plugin depends on
`notify`. So the dependency graph is not something I should be re-typing into a
tsconfig — it is already computed.

So [`@nx/js:typescript-sync`](https://nx.dev/concepts/sync-generators) reads that
graph and writes the references to match it. Run `nx sync` and the tsconfigs are
correct. And — this is the part that turned my build red — it also **gates**: if
the references on disk do not match the graph, nx stops before the tasks run. In
CI that is a failure with the "out of sync" message above; in a terminal it is
the same message as a warning, plus an offer to apply the fix for you.

That is what stopped me. I had added `@dev-blog/release`, imported
`@dev-blog/notify`, and left the references untouched. nx knew the two disagreed
and refused to build a drifted config. One command fixed it:

```bash
nx sync
```

It wrote two lines — a reference to the new plugin in the root tsconfig, and a
reference to `notify` inside the plugin's `tsconfig.lib.json` — and the build
went green.

## Where it lands

The last time this shape came up here it was icon colours in a generated file
nobody reads. The dependency graph is a fact the imports already contain, and the
project references are that fact, copied by hand into a second file. So the copy
is generated, and a gate fails the build when the copy and the original disagree
— which is the only way I was ever going to catch a reference I forgot.

The honest cost: project references earn their keep at scale, and this repo is
not large, so the incremental-build win is mostly theoretical. What is not
theoretical is that the references still have to be right, and one command writes
them from a graph that already exists.
