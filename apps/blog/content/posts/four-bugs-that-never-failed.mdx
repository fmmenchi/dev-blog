I rebuilt this site over the last few weeks, in the open. Four things were broken while I did it, and **none of them ever failed**. No error, no red test, no warning in the console. All four shipped through a green build.

That is what they have in common, and it is why I am writing this down.

## What the site is

An Nx monorepo: one React Router app, server-rendered, running on Cloudflare Workers, plus two small libraries for the design tokens and the components. Tailwind is there, but on top of my own tokens. Its palette and its scales are wiped, so `bg-neutral-800` does not exist and cannot end up in a component by accident. Only semantic roles survive. Radix does anything that needs a keyboard.

## 1. Two icons that were black on black

The GitHub and LinkedIn icons come from simple-icons, and they arrive like this:

```svg
<svg viewBox="0 0 24 24"><path d="M12 .297c-6.63 0-12 5.373…"/></svg>
```

There is no `fill` on that path. The default fill in SVG is **black**.

My SVGR config had a rule for colour: replace `#000` with `currentColor`, so an icon inherits the colour of the text around it. Reasonable rule. It matched nothing, because there is no `#000` in the file. There is no colour in the file at all.

So the icons rendered black, on a near-black background.

They were not invisible, which is part of why it took me so long. A dark shape on a dark surface reads as a smudge, and the eye slides over it without stopping. The icons were in the DOM. They had correct accessible names. The accessibility sweep had nothing to say about them either, because an icon nobody can see is not an accessibility violation.

I fixed the config, not the SVGs. Those files are vendored, so a hand edit would have been undone the next time somebody re-downloaded one. The config now sets `fill="currentColor"` on the root element outright, instead of looking for a colour to replace.

## 2. A border nobody could see

The chips — the little skill labels — looked wrong. Not obviously wrong. Just loose. Grey words sitting on the card with nothing around them.

So I measured. The text was fine: 6.74:1 against the card, comfortably past AA. The border was **1.18:1**.

<Image
  image={invisibleBorder}
  alt="Two panels. On the left, skill badges whose 1px border measures 1.18:1 against the card: the outline is not there, only grey words float. On the right, the same badges with a border at 3.94:1 and a muted fill: they read as chips."
  sizes="(min-width: 56rem) 42.5rem, 100vw"
/>

1.18 is not a near miss. At that ratio the border is not there at all, so the chip had no shape: only text, and an outline I thought I was drawing in a colour I could not tell apart from the surface behind it. I had been looking straight at it for days.

The border is now 3.94:1, with a muted fill behind it. Contrast is a number, and when something looks off it is worth measuring before you start moving things around.

## 3. The site was waiting for Google

I added Lighthouse to the repo, mostly to have a baseline. It said 71. First paint 4.6 seconds. Total blocking time **0 ms**.

Those two numbers have to be read together. Zero blocking time means the browser had nothing to run: no script, no work. It sat there for four and a half seconds and waited.

<Image
  image={waitingForGoogle}
  alt="A request waterfall. Before: the HTML, then a serial chain — fonts.googleapis.com (DNS, TLS, request) for a stylesheet, and only once that has arrived can fonts.gstatic.com begin, for the font itself. First paint at 4.6 seconds, with nothing executing. After: the HTML and a same-origin woff2, first paint at 3.5 seconds."
  sizes="(min-width: 56rem) 42.5rem, 100vw"
/>

It was waiting for the fonts, which came from Google Fonts. That is one line in the head and two round trips in practice: a render-blocking stylesheet from `fonts.googleapis.com`, which the browser has to fetch and parse before it even discovers that the font file itself lives on a second origin, `fonts.gstatic.com`. Two DNS lookups, two TLS handshakes, two requests, all of them standing between the visitor and the first pixel.

The site was never slow. It was blocked, on infrastructure I do not control, to load a typeface.

There is a second problem in the same line of HTML, and it is the one that actually bothers me. The colophon of this site says: no analytics, no cookies, no tracking. Meanwhile every visitor's IP address was being sent to Google, twice, in order to render the page making that promise.

Self-hosting the fonts fixes both. `font-display: swap` then paints the text immediately in a fallback face, instead of leaving it invisible while a file downloads. But swap has a cost of its own: when the real font arrives the layout jumps, because the fallback has different metrics.

The fix is to adjust the fallback. Arial, with its ascent, descent and width overridden so that it occupies the same space the real font is going to occupy. The text appears at once, the real face swaps in, and nothing moves.

Getting there took three attempts. I blamed the hero image first, then `font-display` itself. Both were wrong. What was actually happening is that I had given JetBrains Mono the same Arial fallback as the body text, and Arial is not monospaced. The nav is set in mono, so every time the real font arrived the nav re-wrapped and the whole page jumped 33 pixels. A monospaced font needs a monospaced fallback. Layout shift went from 0.022 to zero.

## 4. The home page was downloading a dropdown it does not have

I added filters to the post list: tags, and a sort order. The sort control is a Radix Select, which I still think is the right call, because a hand-rolled dropdown is the most reliable way I know of making a site unusable with a keyboard.

Radix Select is also heavy. It brings a portal, a focus scope and floating positioning: around 69 KiB, to do a job the browser has a native element for. That is a fair price on the pages that use it.

Every page was paying it. The component library is imported by the root route, so its chunk is shared by everything, and Radix Select was sitting inside that chunk. The home page has no dropdown, and downloaded the dropdown anyway.

React Router already code-splits by route, and it could not help here: the cost was not inside a route, it was inside a chunk shared between routes.

So I lazy-loaded the filter bar with a dynamic import, which is supposed to give it a chunk of its own. It changed nothing. Same 102 KiB chunk, same Radix, same home page. This is the part I did not understand for an embarrassingly long time.

Look at it from where the bundler is standing. The root route imports `Button` from the library's barrel file, the `index.ts` that re-exports every component, `Select` included. The bundler would like to drop `Select`, since nothing on this page uses it. To drop it, it has to delete the import. And deleting an import is only safe if importing that file does nothing on its own. A module is allowed to do things simply by being loaded: register a global, patch a prototype, inject a stylesheet.

The bundler cannot prove that mine does not, so it keeps it. My dynamic import was real, and it was also useless, because `Select` was still being pulled in by the barrel.

The missing piece is one line, and it is a promise the package makes to the bundler:

```json
"sideEffects": ["*.css"]
```

It says: importing any of my modules does nothing by itself, so if you do not use its exports, throw it away. Only then is `Select` reachable from the dynamic import alone, and only then does the split happen.

It has to be that and not `false`, and the difference matters. A CSS import genuinely is a side effect: importing it injects styles, and nothing ever uses its exports. Declaring `false` would have given the bundler permission to delete `import './prose.module.css'` as dead code, and the styles would have disappeared without any error.

Two changes, and neither one does anything without the other. The shared chunk went from 102 KiB to 12 KiB, and Radix stayed exactly where it was.

## What they have in common

None of them failed.

The icon code was correct. The border was a valid colour, spelled correctly, resolving to a real token. The font link was an ordinary `<link>`, the kind that shows up in every tutorial. The import was an ordinary import. Every test was green, and I would have approved all four of those diffs myself, because there is nothing in any of them to object to.

None of them could be caught by looking, either, which is why I stopped relying on looking and wrote checks instead:

- a check that fails the build on any `var(--token)` the theme does not define;
- a check that fails on any generated icon which hardcodes a colour, or never says what colour it is (that code is git-ignored, so nobody was ever going to read it anyway);
- axe over every page, failing the build on a WCAG violation;
- Lighthouse over the production build, which is what caught the fonts.

Code that nobody reads cannot be caught by review. It needs a gate.

## Where it landed

Performance 71 to 82. First paint 4.6s to 3.4s. Layout shift down to zero. Accessibility 100 on every page. Two icons you can see now.

It's still not fast enough. We could do better. Around **114 KiB** of JavaScript ships for pages that are mostly text and barely need any of it. That is the next thing to fix.

The direction is right, though, and from here the changes can stay small.
